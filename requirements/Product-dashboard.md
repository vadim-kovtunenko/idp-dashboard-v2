# Product Dashboard

Связанный документ: [[Product-dashboard-architecture]]

## 1. Назначение документа

Этот документ фиксирует структурированные бизнес-требования для первой версии продуктового дашборда. Цель документа - перевести исходные тезисы в формат, удобный для кодинга: с явным составом виджетов, фильтров, состояниями по умолчанию, правилами отображения и минимальной моделью данных под локальный `JSON`.

## 2. Контекст MVP

- Дашборд запускается локально на `localhost`.
- Источник данных для MVP - локальный `JSON`.
- `API`, база данных, авторизация и автоматизация сбора данных в текущий scope не входят.
- Требования к визуальному дизайну пока не фиксируются; на этом этапе нужен только функциональный каркас.

## 3. Scope MVP

В первую версию входят 5 виджетов:

1. `initiatives_created` - количество заведенных инициатив
2. `initiatives_in_prod` - количество инициатив в ПРОМе
3. `calls_count` - количество вызовов
4. `source_distribution` - распределение по источникам
5. `tickets_created` - количество заведенных тикетов

## 4. Общие правила

### 4.1. Общие правила по данным

- Все вычисления выполняются на фронте на основе локального `JSON`.
- Для линейных графиков с месячной динамикой данные должны быть доступны в разрезе месяцев.
- Для кастомного периода используется агрегация по месяцам.

### 4.2. Общие правила по фильтрации

- Для виджетов `initiatives_created`, `initiatives_in_prod` и `calls_count` используется двухуровневая фильтрация:
  - уровень 1 - направление (`GigaSearch`, `GigaQuery`, `Summarization`);
  - уровень 2 - внутренний фильтр, набор которого зависит от выбранного направления и конкретного виджета.
- Для виджетов `initiatives_created`, `initiatives_in_prod` и `calls_count` должен быть доступен кастомный диапазон дат.
- Кастомный диапазон дат применяется в связке с остальными фильтрами виджета.
- Если кастомный диапазон не задан, используется отображение за последние 12 месяцев.

### 4.3. Состояния по умолчанию

- Для `initiatives_created` по умолчанию выбраны:
  - направление `GigaSearch`;
  - внутренний фильтр `common`;
  - период `последние 12 месяцев`.
- Для `initiatives_in_prod` по умолчанию выбраны:
  - направление `GigaSearch`;
  - внутренний фильтр `common`;
  - период `последние 12 месяцев`.
- Для `calls_count` по умолчанию выбраны:
  - направление `GigaSearch`;
  - внутренний фильтр `common`;
  - период `последние 12 месяцев`.
- Для `source_distribution` по умолчанию выбран сегмент `alpha`.
- Для `tickets_created` по умолчанию выбран `текущий спринт`.

- `direction`: `GigaSearch`, `GigaQuery`, `Summarization`

### 6.4. `source_distribution`

**Заголовок:** Распределение по источникам

**Тип:** круговая диаграмма

**Фильтр:**

- `network_segment`: `alpha`, `sigma`

**Поведение:**

- При выборе сегмента должна отображаться структура источников только для выбранного сегмента.
- По умолчанию выбран сегмент `alpha`.

**Доступные значения для `alpha`:**

- `KA`
- `SberHelp`
- `ECM`
- `IDP Studio`
- `K+`
- `Custom_source`

**Доступные значения для `sigma`:**

- `M-App`
- `ECM`
- `IDP Studio`
- `Custom_source`

### 6.5. `tickets_created`

**Заголовок:** Количество заведенных тикетов

**Тип:** линейный график

**Фильтры:**

- `sprint`: `current`, `previous`

**Поведение:**

- График должен поддерживать просмотр:
  - только текущего спринта;
  - только предыдущего спринта;
  - наложения предыдущего спринта на текущий на одном графике.
- По оси `X` отображаются 14 дней спринта.
- Допустимый формат разметки оси `X`:
  - либо `1-14`;
  - либо дни недели, начиная с понедельника.
- Значения по оси `Y` в исходных тезисах не зафиксированы, поэтому для MVP допускается авторасчет масштаба по данным.

## 7. Разметка страницы

Используется плиточная компоновка.

### 7.1. Первый ряд

- Слева: `initiatives_created`
- Справа: `initiatives_in_prod`
- Оба графика должны быть на одном уровне и визуально выровнены относительно друг друга.

### 7.2. Второй ряд

- `calls_count` располагается под первым рядом.
- График должен быть растянут на общую ширину двух верхних графиков.

### 7.3. Третий ряд

- Слева: `source_distribution`
- Справа: `tickets_created`
- Оба виджета должны быть на одном уровне, выровнены относительно друг друга и размещены под `calls_count`.

## 8. Минимальная модель данных для MVP

Ниже - рекомендуемая минимальная структура данных, от которой удобно стартовать при реализации.

### 8.1. Месячные метрики

Подходит для:

- `initiatives_created`
- `initiatives_in_prod`
- `calls_count`

Рекомендуемые поля одной записи:

- `metric_id` - идентификатор метрики
- `direction` - `GigaSearch`, `GigaQuery`, `Summarization`
- `segment` - внутренний фильтр метрики
- `month` - месяц в формате `YYYY-MM`
- `value` - числовое значение

Пример:

```json
{
  "metric_id": "initiatives_created",
  "direction": "GigaSearch",
  "segment": "common",
  "month": "2026-04",
  "value": 42
}
```

### 8.2. Распределение по источникам

Подходит для:

- `source_distribution`

Рекомендуемые поля одной записи:

- `network_segment` - `alpha` или `sigma`
- `source` - источник
- `value` - числовое значение

Пример:

```json
{
  "network_segment": "alpha",
  "source": "ECM",
  "value": 120
}
```

### 8.3. Данные по тикетам

Подходит для:

- `tickets_created`

Рекомендуемые поля одной записи:

- `sprint` - `current` или `previous`
- `day_index` - номер дня спринта от `1` до `14`
- `weekday_label` - опциональная подпись дня недели
- `value` - числовое значение

Пример:

```json
{
  "sprint": "current",
  "day_index": 1,
  "weekday_label": "Mon",
  "value": 6
}
```

## 9. Критерии готовности MVP

MVP считается соответствующим этим требованиям, если:

1. На странице присутствуют все 5 виджетов.
2. Плиточная компоновка соответствует описанной структуре.
3. Для виджетов `initiatives_created`, `initiatives_in_prod` и `calls_count` работают:
   - переключение направления;
   - переключение внутренних фильтров;
   - кастомный диапазон дат;
   - агрегация по месяцам.
4. Для `source_distribution` работает переключение между `alpha` и `sigma`.
5. Для `tickets_created` можно отдельно показать текущий и предыдущий спринт, а также наложить их на один график.
6. Значения и шкалы осей соответствуют зафиксированной спецификации по каждому виджету.
